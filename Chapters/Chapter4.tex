\section{Compilación como herramienta de verificación} % 1 página

En el desarrollo del firmware y el software se adoptó la compilación no solo como una etapa necesaria para la generación del código ejecutable, sino también como una herramienta activa de verificación. A través del uso de características del lenguaje C++ en su estándar 20, fue posible detectar una amplia variedad de errores en tiempo de compilación. Como consecuencia, se redujo la probabilidad de fallas en tiempo de ejecución (runtime) y se mejoró la confiabilidad general del sistema.

Este enfoque resulta de interés en sistemas embebidos, donde las capacidades de depuración pueden ser limitadas y los errores detectados en campo pueden implicar costos elevados. En este contexto, el compilador actúa como un verificador estático que valida invariantes del diseño antes de que el firmware sea desplegado en el hardware objetivo.

Una de las herramientas principales empleadas con este fin fue el uso extensivo de expresiones evaluables en tiempo de compilación (\texttt{constexpr} y \texttt{consteval}). Mediante la definición de constantes y funciones evaluables durante la compilación, se aseguraron propiedades estructurales del sistema tales como tamaños de buffers, rangos válidos de parámetros y coherencia entre configuraciones internas. Esta estrategia permitió que inconsistencias de configuración se manifestaran como errores de compilación, en lugar de fallas silenciosas durante la ejecución.

Asimismo, se utilizaron mecanismos de verificación estática explícita para imponer restricciones adicionales. A través de aserciones evaluadas en tiempo de compilación (\texttt{static\_assert}), se validaron supuestos de diseño que no dependen del estado dinámico del sistema, como alineaciones de estructuras, correspondencia entre tipos y límites máximos definidos por el hardware. Este tipo de validaciones documenta de forma explícita las hipótesis del diseño y facilita su mantenimiento a largo plazo.

Otra característica central del lenguaje empleada fue el uso de conceptos, introducidos en C++20. Los conceptos permitieron expresar de manera declarativa los requisitos que deben cumplir los tipos utilizados en interfaces genéricas (plantillas). En lugar de aceptar cualquier tipo compatible sintácticamente y delegar la detección de errores a mensajes crípticos del compilador, se definieron contratos claros sobre las capacidades requeridas. De este modo, el uso incorrecto de una API produce diagnósticos precisos y localizados, lo que simplifica tanto el desarrollo como la verificación del firmware.

Finalmente, se aprovechó el carácter fuertemente tipado del lenguaje para modelar dominios específicos del problema. En lugar de representar conceptos distintos mediante tipos primitivos intercambiables, se definieron tipos diferenciados para representar identificadores, índices, unidades lógicas y estados del sistema. Esta decisión redujo la posibilidad de errores por confusión semántica, ya que operaciones inválidas entre conceptos incompatibles quedan prohibidas por el sistema de tipos y son rechazadas durante la compilación.

Este paradigma de programación resultó en una primera línea de defensa frente a errores de diseño e integración. La detección temprana de inconsistencias contribuyó a disminuir la complejidad de los ensayos dinámicos posteriores y fortaleció la robustez del sistema desarrollado.

\section{Pruebas unitarias del firmware} % 2 páginas

Las pruebas unitarias del firmware se orientaron a verificar el comportamiento funcional de los componentes que forman parte directa del flujo de datos principal de la aplicación. Como se mencionó en \ref{subsection:io}, este flujo comprende la recepción de datos crudos desde las interfaces de entrada, su interpretación como mensajes MIDI, la aplicación de reglas de filtrado y, finalmente, el enrutamiento hacia las salidas correspondientes (figura \ref{fig:svc-in-pipe}).

En base a este criterio, se realizaron pruebas unitarias sobre los siguientes componentes:
\begin{itemize}
	\item \texttt{RingBuffer}
	\item \texttt{app::Parser}
	\item \texttt{app::Filter}
	\item \texttt{app::Router}
	\item \texttt{app::Serializer}
\end{itemize}

\subsection{Pruebas de \texttt{RingBuffer}}
La estructura \texttt{RingBuffer} se utiliza para el almacenamiento temporal dentro del flujo de datos del firmware.

Las pruebas unitarias verificaron las propiedades fundamentales de una cola FIFO circular:
\begin{itemize}
	\item Estado inicial del buffer.
	\item Operaciones de lectura y escritura sin wrap-around.
	\item Operaciones de lectura y escritura con wrap-around.
	\item Condiciones límite, como el llenado completo del buffer y el intento de escritura de datos mayor a su capacidad.
\end{itemize}

\subsection{Pruebas de \texttt{app::Parser}}
La clase \texttt{app::Parser} se encarga de interpretar los datos crudos recibidos y convertirlos en mensajes MIDI estructurados.

Las pruebas unitarias verificaron la correcta decodificación de mensajes MIDI, así como el descarte adecuado de bytes de datos aislados. Se evaluó el manejo de secuencias inválidas y la capacidad del parser de reanudar el procesamiento ante la aparición de nuevos bytes de estado. Además, se validó el tratamiento de mensajes incompletos distribuidos en múltiples bloques de entrada.

\subsection{Pruebas de \texttt{app::Filter}}
La responsabilidad de \texttt{app::Filter} es la de implementar las reglas de filtrado configurables que determinan qué mensajes MIDI continúan su procesamiento dentro del flujo de datos.

Las pruebas unitarias validaron el comportamiento por defecto del filtro, en el que todos los mensajes sean aceptados en ausencia de reglas específicas. Posteriormente, se verificó la correcta aplicación de reglas de filtrado parametrizadas por tipo de mensaje y canal. Asimismo, se comprobó la independencia entre los filtros de mensajes de canal y sistema, para verificar que el bloqueo de un mensaje no afecte a otros no relacionados.

\subsection{Pruebas de \texttt{app::Router}}
El componente \texttt{app::Router} es responsable del enrutamiento de los mensajes MIDI hacia las salidas configuradas.

Las pruebas unitarias se enfocaron en validar el estado inicial sin rutas habilitadas, la activación selectiva de salidas individuales y la coexistencia de múltiples rutas dentro de un mismo conjunto de configuración. Además, se verificó la independencia entre distintos conjuntos de rutas, de modo que las modificaciones en uno de ellos no impacten sobre los demás.

\subsection{Pruebas de \texttt{app::Serializer}}
La clase \texttt{app::Serializer} constituye la etapa final del flujo de datos del firmware y se encarga de convertir los mensajes MIDI estructurados en una secuencia de bytes apta para su transmisión.

Las pruebas unitarias verificaron la correcta serialización de los distintos tipos de mensajes MIDI, de forma tal que la secuencia de bytes generada respete el formato definido por la especificación del protocolo. Asimismo, se comprobó la preservación de la información original de estado, canal y datos.

\section{Pruebas de latencia} % 6 páginas

Dado que la función del equipo es direccionar mensajes desde las entradas hacia las salidas, es fundamental que el tiempo de propagación sea el menor posible. Por otro lado, al tratarse de un contexto de ejecución musical, cualquier retardo significativo puede acarrear efectos indeseados en términos de responsividad. En \cite{wessel2002intimate} se menciona un límite de 10 ms de latencia a partir del cual se compromete la sensación de inmediatez y control musical en sistemas interactivos. Este tiempo fue adoptado como criterio de aceptación dura para validar los ensayos de latencia del equipo.

\subsection{Descripción del banco de pruebas}

Para poder realizar las mediciones de latencia se montó un banco de pruebas que permitió monitorear el comportamiento de las entradas y salidas de la placa central. En la figura \ref{fig:test-bench-db} se ilustra un diagrama en bloques con los componentes principales.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.7\textwidth]{./Figures/test-bench-db.png}
	\caption{Diagrama en bloques del banco de pruebas para medir la latencia del equipo.}
	\label{fig:test-bench-db}
\end{figure}

El procedimiento de prueba consiste en tomar el DUT (\textit{Device Under Test}) e inyectarle mensajes MIDI. Para lograr este fin, se omitió la placa de entradas y se conectaron conectores USB-Serie de manera directa. Con ayuda de un analizador lógico, se conectaron sondas en las entradas y salidas de la placa y se midió el tiempo de retardo.

Adicionalmente, se conectaron un programador y un conversor USB-Serie extra para realizar cambios de firmware y monitorear mensajes de debug emitidos por la placa, respectivamente. En la figura \ref{fig:test-bench} se observa el banco de pruebas montado.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.6\textwidth]{./Figures/test-bench.jpg}
	\caption{Fotografía del banco de pruebas montado.}
	\label{fig:test-bench}
\end{figure}

Numerados se encuentran:
\begin{enumerate}
	\item El DUT (la placa central del router).
	\item El analizador lógico.
	\item El programador.
	\item El conversor USB-serie de debug.
	\item Los conversores USB-serie que inyectan mensajes MIDI.
\end{enumerate}

La inyección de mensajes y su captura en el analizador lógico fueron automatizadas mediante scripts de Python. Esto permite generar distintos casos de prueba y recolectar métricas que permitan caracterizar al equipo de forma repetitiva. Para este trabajo se contemplaron los siguientes casos:

\begin{itemize}
	\item Una entrada ruteada a todas las salidas (1$\rightarrow$8).
	\item Todas las entradas ruteadas a una salida distinta, sin merge (4$\rightarrow$4).
	\item Todas las entradas ruteadas a una salida (4$\rightarrow$1).
\end{itemize}
